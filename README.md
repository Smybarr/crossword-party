# World's Largest Crossword - Party App

A collaborative web app for solving the "World's Largest Crossword Puzzle" (28,923 clues!) at crossword parties.

## Background

This project was inspired by a friend (Dan) who hosts crossword puzzle parties using the [QVC World's Largest Crossword Puzzle](https://www.qvc.com/play-view-brands-worlds-largest-crossword-puzzle.product.T59491.html). The physical puzzle is 6.5' x 6.5' with over 91,000 squares. The original setup involved paper copies of clues and scanned PDFs—this app aims to streamline the experience.

**This is a personal pet project, not intended for commercial distribution.**

## Tech Stack

- **React 19** with TypeScript
- **Vite 7** for dev server and builds
- **Tailwind CSS v4** for styling
- **shadcn/ui** component library

## Reference Files

Located in `/ref`:

### Source Data
- `clues_db.json` - Original 28,923 clues in JSON format (Dan created this). **Contains 73 known duplicates** due to OCR errors — see [Known Data Issues](#known-data-issues) below.
- `clues_db_corrected.json` - Corrected version with all OCR fixes applied (page 55 Down renumbering, page 75 Down renumbering, 4453→4452 fix). 28,922 unique clues, zero duplicates. Generated by Claude Code.
- `crossword_clue_book.pdf` - Original 104-page clue book scan
- `grid_layout.csv` - 303×303 grid layout (Dan created this, March 2026). Each cell is `1` (white) or empty (black). Dan built the upper-left quadrant by hand (using AI-written macros for diamond patterns + formulas), then mirrored it via 180° rotation — so any cell error in the upper-left appears in all quadrants. Estimated ~90-95% accurate based on spot-checking against the physical puzzle photo; known errors are isolated white cells that should be black.

### Answer Key
- `answer_key.json` - Structured answer key mapping each clue to its grid position, direction, and entry length. Generated by numbering the grid per standard crossword rules and cross-referencing with `clues_db_corrected.json`. Each entry has a status: `matched` (64.3%) or `direction_mismatch` (35.7%).
- `answer_key.txt` - Human-readable version. Entries tagged `[OK]` (matched), `[??]` (direction mismatch), or `[!!]` (number not in grid).

### Grid Analysis
- `grid_correction_suggestions.html` - Interactive visual overlay of the full 303×303 grid. Open in a browser. See [Using the Grid Correction Tool](#using-the-grid-correction-tool) below for a walkthrough.
- `grid_fixes.csv` - Spreadsheet of suggested cell corrections with row, column, current value, suggested value, and which clue(s) indicate the fix. Derived from an earlier analysis pass; the HTML tool above is more refined and should be preferred for spot-checking.

### Answer Key Scans (Legacy)
- `answer_key/grid{1-4}.jpg` - Four quadrant scans of the physical answer key (~2900×2700px each). Layout: Grid 3 = top-left, Grid 4 = top-right, Grid 2 = bottom-left, Grid 1 = bottom-right.
- `answer_key/grid{1-4}_r{0-1}_c{0-2}.json` - OCR'd answer data from each sub-grid (Dan created these). Letters = white cells, `?` = black cells. **Known to contain OCR errors** — these were superseded by `grid_layout.csv`.

### Grid Layout vs Clue Database Alignment

The grid numbering (24,733 numbered cells) aligns perfectly with the clue database — every clue number maps to the correct grid position. However, ~36% of clues have a **direction mismatch**: the clue says Across but the grid only supports Down at that position (or vice versa).

Key stats:
- 18,586 / 28,922 clues (64.3%) fully match the grid
- 10,336 clues have direction mismatches (roughly even: 5,270 Across-but-only-Down + 5,066 Down-but-only-Across)

#### Root cause analysis

The 10,336 mismatches break down into two categories:

1. **~7,100 likely direction label errors in the clue database.** The grid supports one direction at a numbered cell, the DB says the other, and no conflicting clue exists. Swapping the direction label would fix these. This alone would bring the match rate from 64.3% to ~88.8%.

2. **~3,200 grid cell errors.** The DB has both Across and Down clues for a number, but the grid only supports one direction. A missing black square (white cell that should be black) prevents the other direction's entry from starting. These correspond to the isolated cell errors Dan identified during spot-checking (~5-10% of cells, all WHITE→BLACK).

#### How Dan built the grid

Dan digitized the upper-left quadrant by hand using AI-written spreadsheet macros for the diamond patterns, then mirrored it via 180° rotation to fill all four quadrants. This means any cell error in the upper-left quadrant (rows 0-151, cols 0-151) is duplicated to the other three quadrants. Spot-checking confirmed the macro patterns are correct; errors are isolated individual cells.

#### Next steps

1. **Spot-check the grid** using `grid_correction_suggestions.html` (see [walkthrough below](#using-the-grid-correction-tool)) to fix the ~3,200 cell errors
2. **Fix direction labels** in the clue database for the ~7,100 mismatches (can be done programmatically once the grid is finalized)
3. **Regenerate the answer key** after both corrections are applied

## Using the Grid Correction Tool

Open `ref/grid_correction_suggestions.html` in a browser. This is an interactive canvas-based viewer for spot-checking the grid CSV against the physical puzzle.

### Navigation

- **Scroll** to zoom in/out
- **Click and drag** to pan
- **Hover** over any cell to see its position, clue number, quadrant, symmetric pair, and any suggested fix with reasoning

### Filter controls (top of page)

| Control | What it does |
|---------|-------------|
| **Show flip-to-BLACK** | Toggle red highlights (white cells that may need to be black) |
| **Show flip-to-WHITE** | Toggle green highlights (black cells that may need to be white) |
| **High confidence only** | Show only fixes where both Across AND Down clues exist in the DB for that number — these are the most trustworthy suggestions because the clue data independently confirms something is wrong at that grid position |
| **Highlight UL quadrant** | Draw a purple border around the upper-left quadrant (rows 0-151, cols 0-151) — this is the source quadrant Dan built by hand; fixes here propagate to all quadrants |
| **Show clue numbers** | Display clue numbers on numbered cells (visible when zoomed in) |
| **Show symmetric pairs** | Click a cell to highlight it (yellow) and its 180° mirror (cyan) |

### Recommended workflow

1. Check **"High confidence only"** and **"Highlight UL quadrant"**
2. Zoom into the upper-left quadrant and look for red cells (suggested flip-to-BLACK)
3. Compare each red cell against the physical puzzle or a photo of the grid
4. Focus on one row or region at a time — the hover tooltip shows which clue(s) suggest each fix
5. Once high-confidence fixes are verified, uncheck "High confidence only" to see all suggestions

### Confidence levels

- **High confidence** (solid colors): Both an Across and Down clue exist for this number in the database, but the grid only supports one direction. The clue data strongly suggests this cell is wrong.
- **Low confidence** (faded colors): Only one clue direction exists in the DB and it doesn't match the grid. This could be a grid error OR a direction label error in the clue database — needs manual verification.

**Any code changes pushed to the `main` branch will automatically trigger a new production deployment.** This means broken code on `main` = broken live app. To avoid this:

- **Use a feature branch** for any non-trivial changes. Create a branch (`git checkout -b my-feature`), make your changes, test locally with `npm run dev`, then open a pull request to merge into `main`.

- **Keep `main` stable** — only merge when you're confident the change works.

If the repo ever has more contributors and you want stronger safeguards, you can enable [branch protection rules](https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/managing-a-branch-protection-rule/about-branch-protection-rules) on `main` to require pull request reviews before merging.

## Authentication

The app uses Supabase magic link auth. Users sign up with their email, first name, and last name, then receive a magic link to log in. Profiles are stored in a `profiles` table and the generated display name (e.g. "John D.") is used for the leaderboard and solve attribution.

**Email case sensitivity**: Emails are handled case-insensitively throughout. The `auto-signin` edge function lowercases both sides when looking up existing users, so `John@example.com` and `john@example.com` resolve to the same account. Returning users are signed in instantly without a magic link email; new users are prompted for their name and sent a magic link to complete registration.

## Known Data Issues

The source `clues_db.json` (28,923 entries) contained 73 duplicate `(number, direction)` pairs with conflicting clue text. Investigation by cross-referencing against the original PDF clue book identified three root causes, all related to OCR/data-entry errors during digitization:

### 1. Page 55 Down clues 825–997 (68 duplicates + 27 non-duplicate errors)

**Cause**: 95 Down clues on page 55 with numbers 825–997 were actually clues **1825–1997** — the leading "1" was dropped during OCR. These overlapped with the legitimate page 53 Down clues in the same number range.

**Resolution**: Renumbered all page 55 Down clues in the 825–997 range by adding 1000 (e.g., 825 → 1825). This also filled a gap of missing clues in the 1825–1997 range.

### 2. Page 75 Down clues 21, 23, 24 (3 duplicates)

**Cause**: Three Down clues on page 75 had truncated numbers — they were actually clues **12121, 12123, 12124** (the leading "121" was dropped). These overlapped with the legitimate page 51 Down clues 21 and 24.

**Resolution**: Renumbered to 12121, 12123, and 12124 respectively.

### 3. Page 59 Down clue 4453 (1 duplicate)

**Cause**: Clue 4453 Down appeared twice on page 59 with different text. Positional analysis showed the first entry ("Auditory challenged") was actually **Down 4452** — the clue number was misread.

**Resolution**: Renumbered the first occurrence to 4452. "Bellicose god" remains as 4453.

### Final count

After corrections: **28,850 unique clues** (up from 28,849 after deduplication, since the 4452 clue was recovered). Zero remaining duplicates.

## Keeping the Database Alive Between Parties

The Supabase Free Plan **pauses projects after 7 days of inactivity** (no API requests). Since crossword parties may be 2-3 months apart, expect the database to pause between events.

**Your options:**

1. **Just let it pause and restore before the next party** — Click "Restore" in the [Supabase dashboard](https://supabase.com/dashboard) a few days before the party. It takes a few minutes and all data is preserved. You have a **90-day window** for one-click restore after pausing.

2. **Ping it periodically** — A cron job, GitHub Action, or any scheduled task that hits the Supabase API once a week will prevent auto-pause. Even a simple `SELECT 1` via the REST API counts as activity.

3. **Upgrade to Pro ($25/mo)** — Pro projects never auto-pause. Likely overkill for this use case.

For a party app with 2-3 month gaps, option 1 is the simplest. The 90-day restore window comfortably covers the downtime.
