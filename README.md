# World's Largest Crossword - Party App

A collaborative web app for solving the "World's Largest Crossword Puzzle" (28,923 clues!) at crossword parties.

## Background

This project was inspired by a friend (Dan) who hosts crossword puzzle parties using the [QVC World's Largest Crossword Puzzle](https://www.qvc.com/play-view-brands-worlds-largest-crossword-puzzle.product.T59491.html). The physical puzzle is 6.5' x 6.5' with over 91,000 squares. The original setup involved paper copies of clues and scanned PDFs—this app aims to streamline the experience.

**This is a personal pet project, not intended for commercial distribution.**

## Tech Stack

- **React 19** with TypeScript
- **Vite 7** for dev server and builds
- **Tailwind CSS v4** for styling
- **shadcn/ui** component library
- **Supabase** for database, real-time subscriptions, and auth (magic link)
- **canvas-confetti** for solve celebrations

## Reference Files

Located in `/ref`:

### Source Data
- `clues_db.json` - Original 28,923 clues in JSON format (Dan created this via Gemini OCR). **Contains OCR errors** — see [Known Data Issues](#known-data-issues) below. Kept as a backup/reference.
- `clues_db_corrected.json` - Corrected version with all OCR fixes applied (duplicates resolved, missing clues added, mashed-together clues split, phantom clue removed). 28,928 unique clues, zero duplicates.
- `crossword_clue_book.pdf` - Original 104-page clue book scan
- `grid_layout.csv` - 303×303 grid layout. Each cell is `1` (white) or empty (black). Verified correct as of March 2026 — all 28,928 clue positions match the clue database with zero discrepancies.

### Answer Key
- `answer_key.json` - Structured answer key mapping each clue to its grid position, direction, and entry length. Generated by joining `clue_positions.csv` with `clues_db_corrected.json`. All 28,928 clues matched to the grid (100%).

### Utility Scripts
- `get_clues.py` - Computes clue numbering and positions from the grid layout (generates `clue_positions.csv`)
- `find_divergence.py` - Diffs `clue_positions.csv` against `clues_db_corrected.json` to find any discrepancies
- `generate_answer_key.py` - Joins `clue_positions.csv` + `clues_db_corrected.json` into `answer_key.json`

### Generated Data
- `clue_positions.csv` - 28,928 clue positions with number, direction, row, column, and word length. Generated by `get_clues.py`.

### Grid Photos
- `grid_photos/grid{1-4}.jpg` - Four quadrant photos of the physical puzzle (~2900×2700px each). Useful for visual reference when building the digital crossword interface.

### Grid Layout vs Clue Database Alignment

The corrected grid and clue database are in perfect alignment — all 28,928 clues map to valid grid positions with zero discrepancies. Run `python3 ref/find_divergence.py` to verify.

Vercel auto-detects the Vite preset. The only configuration needed:

**Environment variables** (set in Vercel project settings):

| Variable | Description |
|----------|-------------|
| `VITE_SUPABASE_URL` | Supabase project URL |
| `VITE_SUPABASE_ANON_KEY` | Supabase anon/publishable key |

**Any code changes pushed to the `main` branch will automatically trigger a new production deployment.** This means broken code on `main` = broken live app. To avoid this:

- **Use a feature branch** for any non-trivial changes. Create a branch (`git checkout -b my-feature`), make your changes, test locally with `npm run dev`, then open a pull request to merge into `main`.
- **Vercel generates preview deployments** for every pull request, so you can test changes at a temporary URL before merging to production.
- **Keep `main` stable** — only merge when you're confident the change works.

If the repo ever has more contributors and you want stronger safeguards, you can enable [branch protection rules](https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/managing-a-branch-protection-rule/about-branch-protection-rules) on `main` to require pull request reviews before merging.

## Authentication

The app uses Supabase magic link auth. Users sign up with their email, first name, and last name, then receive a magic link to log in. Profiles are stored in a `profiles` table and the generated display name (e.g. "John D.") is used for the leaderboard and solve attribution.

**Email case sensitivity**: Emails are handled case-insensitively throughout. The `auto-signin` edge function lowercases both sides when looking up existing users, so `John@example.com` and `john@example.com` resolve to the same account. Returning users are signed in instantly without a magic link email; new users are prompted for their name and sent a magic link to complete registration.

## Known Data Issues

The source `clues_db.json` (28,923 entries) contained several categories of OCR errors from the initial Gemini scan of the clue book. These were identified by cross-referencing against the original PDF clue book and diffing computed grid positions against the clue database.

### 1. Duplicate clue numbers (73 duplicates)

**Page 55 Down clues 825–997** (68 duplicates): 95 Down clues had numbers 825–997 but were actually **1825–1997** — the leading "1" was dropped during OCR. **Resolution**: Renumbered by adding 1000.

**Page 75 Down clues 21, 23, 24** (3 duplicates): Three clues had truncated numbers — actually **12121, 12123, 12124**. **Resolution**: Renumbered.

**Page 59 Down clue 4453** (1 duplicate): Appeared twice with different text. First entry was actually **4452**. **Resolution**: Renumbered.

### 2. Mashed-together clues

Several adjacent clues were concatenated into a single entry during OCR:
- "Runs fast: Br. cit" → split into 21643 Across ("Runs fast: Br.") and 21645 Across ("_____ cit")
- "TV's \"Law and ____\" for Love\": Dryden drama" → split into 7308 Down and 7310 Down
- "Bridget's being ____ up (go formal)" → split into 20912 Down and 20913 Down
- "Bad to the" → completed to "Bad to the _____"

### 3. Missing clues

Five clues were absent from the OCR output and added manually: 21512 Across ("_____-fi"), 24059 Across ("______ and hounds"), 1829 Down ("Chopin composition"), 12304 Down ("Muslim god"), and 21645 Across (split from mashed entry above).

### 4. Number errors

Two clue numbers were off by one: 21208→21209, 21438→21439. One text error: "Vittoria town"→"Witchhunt town" (12463 Down).

### 5. Phantom clue

2954 Down ("Love Bug, e.g.") does not correspond to any position on the physical puzzle grid. Removed.

### Final count

After all corrections: **28,928 unique clues** (14,464 Across + 14,464 Down). Zero duplicates, zero discrepancies with the grid.

## Keeping the Database Alive Between Parties

The Supabase Free Plan **pauses projects after 7 days of inactivity** (no API requests). Since crossword parties may be 2-3 months apart, expect the database to pause between events.

**Your options:**

1. **Just let it pause and restore before the next party** — Click "Restore" in the [Supabase dashboard](https://supabase.com/dashboard) a few days before the party. It takes a few minutes and all data is preserved. You have a **90-day window** for one-click restore after pausing.

2. **Ping it periodically** — A cron job, GitHub Action, or any scheduled task that hits the Supabase API once a week will prevent auto-pause. Even a simple `SELECT 1` via the REST API counts as activity.

3. **Upgrade to Pro ($25/mo)** — Pro projects never auto-pause. Likely overkill for this use case.

For a party app with 2-3 month gaps, option 1 is the simplest. The 90-day restore window comfortably covers the downtime.
